CI/CD Pipeline

This page documents how code moves from GitHub into the SchoolCloud AWS environment, including Terraform automation, security checks, and deployment history.

GitHub Actions Terraform OIDC to AWS Security Scans

CI/CD Architecture

Developers push changes to GitHub. GitHub Actions runs a Terraform Plan workflow and a Security Scan workflow on pull requests, then a Deploy workflow on the main branch. The Deploy job runs Terraform apply, syncs the web app in apps/web to the S3 web bucket, and invalidates CloudFront so the latest version of the SchoolCloud portal is served to users.

CI/CD pipeline diagram exported from Figma (apps/web/img/cicd-pipeline-architecture.png).

Pipeline flow

  1. Developer opens a feature branch and creates a pull request to the secure-serverless-portal repository.
  2. The Terraform Plan workflow runs on the pull request: it checks out the code, sets up Terraform, configures AWS via OIDC, and runs terraform fmt, validate, and plan on the infra/ configuration.
  3. The Security Scan workflow also runs on the pull request: tfsec and checkov scan the Terraform code for misconfigurations and policy violations, and surface findings in the GitHub Actions logs.
  4. After review, the pull request is approved and merged into main.
  5. The Deploy workflow runs on the main branch: Terraform init/plan/apply updates the AWS environment, then the job syncs apps/web to the S3 web bucket and triggers a CloudFront invalidation so the new portal version is live.
